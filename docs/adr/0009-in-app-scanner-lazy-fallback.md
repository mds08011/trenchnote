# ADR 0009 — In-app scanner: native detector first, lazy vendored fallback

**Status:** accepted · **Date:** 2026-07-09

## Context

The native camera app is TrenchNote's primary scan path by design — a tag
is just a URL (ADR 0001). But it falls short in three real situations:
scanning the next tag while already inside the app, iOS camera apps that
are fussy about small/dirty codes, and rapid-fire inventory walks where
bouncing out to the camera app per tag is unusable.

## Decision

- **A dedicated `scan.html`** (one page per question: "what am I looking
  at?"), linked from every header. Single-scan mode decodes and navigates
  to the asset page. The parser accepts TrenchNote URLs from **any**
  origin, so labels printed against an old LAN IP keep working in-app
  after a domain move.
- **Decode with the native `BarcodeDetector`** where it genuinely supports
  QR (feature-detected via `getSupportedFormats()`, not just API
  presence) — Chrome/Android, the field majority, zero bytes.
- **Fallback: vendored jsQR (Apache-2.0, ~130 KB min / ~45 KB wire)** for
  browsers without it (iOS Safari). It is **not in the service worker
  precache and not referenced by any script tag** — scan.html injects it
  only when the camera opens and BarcodeDetector is absent. To keep iOS
  offline scanning possible without precaching for everyone, the service
  worker's shell handler runtime-caches successful static fetches, so the
  fallback self-caches on the phones that actually needed it. Honest
  limitation: an iOS phone that has never opened the scanner online can't
  scan offline.
- **Walk mode: the location selector is the mode switch.** "I'm at: Main
  Yard" keeps the camera open; every scanned tag lands on a running list
  with a verdict — ✓ here, ⚠ "ledger says X", "not in the system".
  Repeat scans bump the existing row instead of duplicating. **Mismatches
  get a one-tap "Move here"**: the walker standing next to the physical
  thing at a declared location is the highest-confidence correction the
  system ever receives, and it's an ordinary ledger-first move with a
  pre-generated id — offline-queueable like every other write.
- Walk mode is assets-only (bulk items have no tags) and read-only apart
  from that explicit button.
- Failures are words, not console errors: camera denial gets a plain
  panel that also reminds people the camera app still works; a foreign QR
  (someone's Wi-Fi code) gets "That's not a TrenchNote tag" and scanning
  continues.

## Consequences

- Chrome/Android phones never download a byte of decoder library; the
  scan page itself is ~13 KB of source.
- The decode loop runs ~6 fps — plenty for a handheld tag, kind to old
  phones and batteries; the camera is released on page-hide.
- Verification in development drives the decode pipeline with QR codes
  generated by the vendored label library (both detector paths tested);
  real-camera behavior (focus, glare, mud) rides on the maintainer's
  field tests, stated in the commit.
- Rejected: zxing (10× the weight for formats TrenchNote never scans),
  precaching the fallback (ships iOS's weight to everyone), and a scanner
  modal inside index.html (duplicates the moment a second page wants it).

# TrenchNote domain model

**Authority:** Descriptive for current entities; proposed for future ecosystem
concepts

**Reviewed:** 2026-07-12

This document describes the model as implemented. It does not redesign the
schema for symmetry, and it does not make proposed ecosystem contracts current.
PocketBase migration files remain the source of the database shape.

## Identity conventions

**CURRENT:** every PocketBase record has a local opaque record `id`. The public
API exposes it, but TrenchNote has not declared that local ID to be a permanent
cross-application identifier.

Business identifiers are narrower:

- `assets.tag_code` is unique and permanent once printed. It identifies one
  physical asset in field use and is part of the QR contract.
- `items.item_code` is optional and non-unique. It is a catalog/reference code,
  not a universal identifier.
- `locations.job_code` is optional accounting text associated with a location,
  not a project entity or globally scoped project identifier.

**UNKNOWN:** no committed organization ID, project public ID, ecosystem object
namespace, or external-reference registry exists.

## Current entities

### Item

| Property | CURRENT model |
| --- | --- |
| Purpose | Describes what a kind of thing is; branches into `unique` or `bulk` tracking |
| Authoritative owner | TrenchNote catalog |
| Key identifier | PocketBase `id` locally |
| Human identifier | `name`; optional non-unique `item_code` |
| Mutability | Authenticated create/update; superuser-only delete |
| Lifecycle | Created, edited as catalog knowledge changes, optionally referenced indefinitely |
| Relationships | Unique items have assets; bulk items appear directly on movements |
| Evidence | Optional catalog photo |
| Import/export relevance | Included in API contract; no dedicated catalog export/import workflow |
| Status | **CURRENT** |

`tracking_mode` distinguishes unique assets from bulk quantities. `meter`
states whether assets of this kind use hour or odometer observations.

### Location

| Property | CURRENT model |
| --- | --- |
| Purpose | Names a jobsite, yard, warehouse, or transit place where subjects can be held |
| Authoritative owner | TrenchNote reference data |
| Key identifier | PocketBase `id` locally |
| Human identifier | `name`; optional `job_code` is accounting context, not identity |
| Mutability | Authenticated create/update; superuser-only delete |
| Lifecycle | Created and maintained by operators; referenced by movements and asset cache |
| Relationships | Origin/destination of movements; target of `assets.current_location` |
| Evidence | None |
| Import/export relevance | Included in API contract; candidate source for a future project-reference mapping |
| Status | **CURRENT** |

`notify_email` is an optional side-effect destination. It is not movement
authority and does not record mail delivery state.

### Asset

| Property | CURRENT model |
| --- | --- |
| Purpose | Represents one physical instance of a uniquely tracked item |
| Authoritative owner | TrenchNote asset registry |
| Key identifier | PocketBase `id` locally |
| Human identifier | Unique permanent `tag_code`; optional serial number |
| Mutability | Authenticated create/update; superuser-only delete |
| Lifecycle | Registered, labeled, moved, optionally rented/assigned/read/inspected; no retirement state exists |
| Relationships | Belongs to item; caches current location; has movements, reservations, readings, requirements, and inspections |
| Evidence | Evidence is attached to related ledgers, not directly to the asset |
| Import/export relevance | `tag_code` is the strongest current handoff candidate; asset records are API contract |
| Status | **CURRENT** |

Ownership is `owned` or `rented`. Vendor, typed PO reference, on-rent date, and
off-rent date are optional facts. Commercial rates are deliberately absent.
`assigned_to` is free-text custody. `current_location` is a cache, not the
authoritative movement record.

### Movement

| Property | CURRENT model |
| --- | --- |
| Purpose | Records an asset move or bulk receipt, transfer, or consumption |
| Authoritative owner | TrenchNote movement ledger |
| Key identifier | PocketBase `id`, pre-generated by first-party clients for idempotency |
| Human identifier | None; display uses subject, locations, person text, and timestamp |
| Mutability | Create-only for authenticated clients; administrative superuser access remains possible |
| Lifecycle | Appended once; corrected by another movement, not an edit |
| Relationships | Exactly one asset or item; optional origin; shape-dependent destination |
| Evidence | Note, mover text, and optional receiving vendor/PO/OS&D text, packing slip, and photos |
| Import/export relevance | Core logistics fact in API contract; no versioned movement-export envelope exists |
| Status | **CURRENT** |

Valid shapes:

| Shape | Subject | Origin | Destination | Quantity |
| --- | --- | --- | --- | --- |
| Asset move | Asset | Optional | Required | `0` |
| Bulk receive | Item | Empty | Required | Positive integer |
| Bulk transfer | Item | Required | Required | Positive integer |
| Bulk consume | Item | Required | Empty | Positive integer |

Receiving fields are optional and not server-restricted to receive shape. The
first-party UI only offers them in delivery mode.

### Reservation

| Property | CURRENT model |
| --- | --- |
| Purpose | Records a human claim on a specific asset |
| Authoritative owner | TrenchNote reservation records |
| Key identifier | PocketBase `id` |
| Human identifier | Requester text plus needed/release dates |
| Mutability | Authenticated create/update; superuser-only delete |
| Lifecycle | Empty/`open` → `fulfilled` or `cancelled`; people choose the transition |
| Relationships | Belongs to one asset |
| Evidence | Free-text note only |
| Import/export relevance | Included in API contract; no dedicated export |
| Status | **CURRENT** |

Empty status is intentionally interpreted as open for rows created before the
status field existed. Reservations are claims, not dispatch orders.

### Reading

| Property | CURRENT model |
| --- | --- |
| Purpose | Records an hour-meter or odometer observation on one asset |
| Authoritative owner | TrenchNote reading ledger |
| Key identifier | PocketBase `id`, pre-generated by first-party clients |
| Human identifier | Asset tag, reading type/value, and date |
| Mutability | Create-only for authenticated clients; administrative superuser access remains possible |
| Lifecycle | Appended once; corrections are later observations |
| Relationships | Belongs to one asset; reading type is copied from the item at capture |
| Evidence | Optional gauge photo and recorder text |
| Import/export relevance | Included in API contract; useful to optional billing analysis without putting rates in core |
| Status | **CURRENT** |

`read_at` is the optional observation date. Latest reading is derived by
`read_at`, falling back to `created`. A decrease is retained and flagged by the
UI rather than rejected or stored as an error state.

### Inspection requirement

| Property | CURRENT model |
| --- | --- |
| Purpose | Describes one recurring inspection obligation for one asset |
| Authoritative owner | TrenchNote inspection catalog |
| Key identifier | PocketBase `id` |
| Human identifier | Requirement name and free-text reference |
| Mutability | Authenticated create/update; superuser-only delete |
| Lifecycle | Added and revised as the operator's program changes |
| Relationships | Belongs to one asset; may be referenced by inspections |
| Evidence | Citation text only; source program is external |
| Import/export relevance | Included in API contract and inspection CSV context |
| Status | **CURRENT** |

`interval_days` is the only scheduling value. The record does not assign work,
approve a standard, or prove legal compliance.

### Inspection

| Property | CURRENT model |
| --- | --- |
| Purpose | Records a pass, fail, or removal observation on a physical asset |
| Authoritative owner | TrenchNote inspection ledger |
| Key identifier | PocketBase `id`, pre-generated by first-party clients |
| Human identifier | Asset tag, requirement/name, inspector text, and date |
| Mutability | Create-only for authenticated clients; administrative superuser access remains possible |
| Lifecycle | Appended once; corrections are later observations |
| Relationships | Belongs to asset; optional requirement must belong to same asset |
| Evidence | Note and optional photo |
| Import/export relevance | API contract and paginated client-side CSV export |
| Status | **CURRENT** |

`inspected_at` is the field observation date; `created` is server entry time.
RED/YELLOW/GREEN standing is derived and is not a field on this entity.

## Current derived concepts

| Concept | Derivation | Status |
| --- | --- | --- |
| Asset location | Latest intended asset movement, with `assets.current_location` used as a cache | **CURRENT** |
| Bulk stock by location | Quantities received/transferred in minus transferred/consumed out | **CURRENT** |
| Bulk total | External receipts minus consumptions | **CURRENT** |
| Current job | Asset's current location → `job_code` | **CURRENT** |
| Latest reading | Newest `read_at`, then `created`, for an asset | **CURRENT** |
| Reading decrease warning | Compare consecutive derived reading order | **CURRENT** |
| Inspection next due | Latest passing observation plus `interval_days` | **CURRENT** |
| Asset inspection badge | Worst derived requirement/inspection condition | **CURRENT** |
| Open reservation | `status` is empty or `open` | **CURRENT** |

## Concepts not present in the current model

- Organization or tenant
- Project as a first-class record
- Stable ecosystem-wide public identifier
- External-reference/provenance registry
- Unit-of-measure field for bulk quantities
- Installation, acceptance, startup, or turnover record
- Signature or signatory identity
- Frozen snapshot or record lock
- Evidence hash or evidence envelope
- Correction/supersession link between ledger rows
- Lifecycle event or handoff manifest
- Import job and import provenance

Absence is important: documentation must not imply these concepts exist merely
because proposed contracts below name them.

## Proposed ecosystem concepts

The following are **PROPOSED**, non-binding concepts for cross-application
coordination. They are not requests to add TrenchNote collections.

| Concept | Proposed purpose | Likely owner |
| --- | --- | --- |
| Project reference | Unambiguous source-system project identity plus optional human code | Each product owns its project; handoff carries the reference |
| Public identifier | Stable namespaced ID distinct from a local PocketBase record ID | Issuing product |
| External reference | Link a local record to an object in another system without copying authority | Consuming product, with issuer provenance |
| Evidence envelope | Describe a file, observation, hash, media type, and provenance consistently | Product that stores the evidence |
| Lifecycle event | Versioned statement that an authoritative domain fact occurred | Product authoritative for the fact |
| Handoff manifest | Bounded package of references and evidence transferred between contexts | Producing product; receipt recorded by consumer |
| Import provenance | Source, version, time, actor, and original identifiers for imported data | Consuming product |

A dedicated ecosystem-contract draft is the next documentation-baseline task.
No application code should conform to that future draft until its open
questions are resolved and an ADR accepts a contract version.
